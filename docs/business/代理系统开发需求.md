# 代理 Key 管理系统 — 开发需求文档

## 一、背景

在现有 API Key 体系基础上，新增"代理"角色。代理拥有主 Key，可自行创建子 Key 分配给客户，实现二级分销。

## 二、数据结构改动

### `ApiKeyInfo` 新增字段（`config/config.go`）

```go
type ApiKeyInfo struct {
    // ... 现有字段保持不变 ...

    // === 新增：代理系统 ===
    Role        string  `json:"role,omitempty"`        // "", "agent"
    ParentKeyID string  `json:"parentKeyId,omitempty"` // 子 Key 指向父 Key 的 ID
    Discount    float64 `json:"discount,omitempty"`    // 代理折扣 0.88 = 88折
}
```

- `Role="agent"` 的 Key 是代理主 Key
- `ParentKeyID` 非空的 Key 是子 Key，值为其代理主 Key 的 ID
- 普通用户 Key 这两个字段都为空，完全不受影响
- 所有字段 `omitempty`，向后兼容

### 关系图

```
Config.ApiKeys[]
  ├── 普通 Key（Role="", ParentKeyID=""）
  ├── 代理主 Key（Role="agent", Discount=0.88）
  │     ├── 子 Key A（ParentKeyID="主Key.ID"）
  │     ├── 子 Key B（ParentKeyID="主Key.ID"）
  │     └── 子 Key C（ParentKeyID="主Key.ID"）
  └── 另一个普通 Key
```

## 三、新增 config 函数

在 `config/config.go` 中新增：

```go
// 查找代理的所有子 Key
func FindSubKeys(parentID string) []ApiKeyInfo

// 代理创建子 Key（生成新 Key，设置 ParentKeyID）
func CreateSubKey(parentID, note string) (*ApiKeyInfo, error)

// 余额划转（从 fromKey 扣 amount，加到 toKey）
func TransferBalance(fromKeyID, toKeyID string, amount float64) error
```

## 四、新增 API 接口

所有接口挂在 `/user/api/agent/*`，用代理主 Key 做 Bearer Token 鉴权。

### 4.1 路由（`proxy/handler_user.go`）

在 `handleUserAPI` 的 switch 中新增：

```go
case strings.HasPrefix(path, "/user/api/agent/"):
    if keyInfo.Role != "agent" {
        writeJSON(w, 403, map[string]string{"error": "agent role required"})
        return
    }
    h.handleAgentAPI(w, r, keyInfo)
```

### 4.2 接口列表

新建 `proxy/handler_agent.go`：

| 路由 | 方法 | 功能 | 请求体 |
|------|------|------|--------|
| `/user/api/agent/summary` | GET | 代理总览 | - |
| `/user/api/agent/subkeys` | GET | 子 Key 列表 | - |
| `/user/api/agent/subkeys` | POST | 创建子 Key | `{"note":"客户A"}` |
| `/user/api/agent/subkeys/{id}` | PUT | 修改子 Key | `{"note":"...", "enabled":true}` |
| `/user/api/agent/subkeys/{id}` | DELETE | 删除子 Key | - |
| `/user/api/agent/subkeys/{id}/topup` | POST | 给子 Key 充值 | `{"amount": 100.0}` |
| `/user/api/agent/subkeys/{id}/usage` | GET | 子 Key 用量 | - |

### 4.3 接口详情

#### GET `/user/api/agent/summary`
```json
{
  "mainKeyBalance": 1500.00,
  "totalSubKeys": 12,
  "enabledSubKeys": 10,
  "totalConsumedBySubKeys": 3200.50,
  "discount": 0.88
}
```

#### GET `/user/api/agent/subkeys`
返回该代理下所有子 Key 列表（不暴露 key 全文，只显示前 8 位 + `***`）。

#### POST `/user/api/agent/subkeys`
创建子 Key，生成新的 `sk-xxx` Key，`ParentKeyID` 设为代理主 Key ID。

#### POST `/user/api/agent/subkeys/{id}/topup`
从代理主 Key 余额划转到子 Key：
1. 检查子 Key 的 `ParentKeyID` 是否为当前代理（**防止越权**）
2. 检查代理余额是否充足
3. 调用 `TransferBalance`

## 五、计费逻辑（不需要改）

核心点：**子 Key 扣费走和普通 Key 完全一样的流程**。

- 子 Key 有自己的 `Balance`
- API 请求时走 `PreAuthorize` → `Reconcile`，扣的是子 Key 的余额
- 代理折扣只在管理员给代理充值时体现（管理员手动操作或未来自动化）

`billing.go` 不需要改动。

## 六、管理员后台改动

### `handler_admin.go`

#### 创建 Key 时
`apiCreateApiKey` 支持传入 `role` 和 `discount`：
```json
POST /admin/api/keys
{
  "note": "代理张三",
  "role": "agent",
  "discount": 0.88
}
```

#### 编辑 Key 时
`apiUpdateApiKey` 支持修改 `role` 和 `discount`。

#### Key 列表展示
`apiGetApiKeys` 返回时附带 `role`、`discount`、`parentKeyId`、`subKeyCount`（子 Key 数量）。

## 七、安全要点

1. **越权检查** — 代理只能操作 `ParentKeyID == 自己ID` 的子 Key
2. **子 Key 不能创建子 Key** — 只有 `Role=agent` 才能创建
3. **删除代理时清理子 Key** — 管理员删除代理主 Key 时，一并删除其所有子 Key（或禁用）
4. **子 Key 不暴露代理信息** — 子 Key 调用 `/user/api/me` 时不返回 `parentKeyId`
